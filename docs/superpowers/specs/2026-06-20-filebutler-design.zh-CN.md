# FileButler 设计计划

日期：2026-06-20

## 目标

将 FileButler 构建为一个可自托管的文件管理工具，支持 Docker 部署或 Linux 本地部署。第一版重点提供安全的网页双栏文件管理器，以及类似 PowerRename 的批量重命名流程。

工具第一阶段面向单管理员使用，同时在账号、操作者和审计边界上为未来的多用户权限模型预留扩展空间。

- PowerRename 功能参考：https://learn.microsoft.com/zh-cn/windows/powertoys/powerrename
- Windows 逻辑字符串比较参考：https://learn.microsoft.com/en-us/windows/win32/api/shlwapi/nf-shlwapi-strcmplogicalw

## 产品范围

### 第一版包含

- Docker 部署。
- Linux 本地二进制部署。
- 首次运行时初始化管理员账号，允许自定义用户名和密码。
- 初始化后的管理员登录。
- 多个已配置的存储根目录。
- 双栏 Web UI。
- 只能在已配置根目录内浏览目录。
- 移动、复制、软链接、硬链接、创建目录和删除操作。
- 批量操作的 dry-run 预览。
- 后台作业执行，包含进度、取消、失败明细和审计日志。
- PowerRename 风格的批量重命名流程。
- 文件列表、重命名预览、枚举编号和执行顺序使用 Windows Explorer 风格的自然排序。
- 使用 SQLite 持久化账号、会话、作业、审计和撤销元数据。

### 第一版明确不包含

- 多用户权限管理。
- 云存储服务。
- 全文搜索。
- 媒体预览。
- 在线文件编辑。
- 高可用或多实例部署。
- Windows 和 macOS 本地部署支持。

## 架构

FileButler 采用单个 Go 服务加 React/Vite 前端。

Go 服务提供：

- REST API。
- 认证和会话处理。
- 文件系统浏览和操作。
- 批量重命名规划和执行。
- 后台作业执行。
- SQLite 持久化。
- 托管已构建的前端静态资源。

React 前端提供：

- 双栏文件浏览。
- 选择源文件和目标目录的工作流。
- 批量操作 dry-run 预览。
- 批量重命名规则编辑和预览。
- 作业进度和失败信息视图。

这种设计保持部署简单：Linux 本地部署只需要一个二进制文件，Docker 部署只需要一个容器镜像。

## 配置

服务启动时读取配置文件。

核心配置：

- 监听地址。
- SQLite 数据库路径。
- 存储根目录列表，每个根目录包含稳定的 root id、显示名称和绝对路径。
- 作业并发数。
- 日志级别。
- 会话 cookie 设置。

Docker 部署时，存储根目录应映射到挂载的 volume。Linux 本地部署时，存储根目录是宿主机文件系统上的绝对路径。

前端绝不发送可信的绝对路径。API 请求使用 `rootId + relativePath` 标识文件。后端负责解析和校验所有路径。

## 首次运行管理员设置

当 SQLite 数据库中没有管理员账号时，服务进入初始化模式。

初始化模式只开放：

- 健康检查。
- 前端静态资源。
- 创建管理员接口。

管理员可以选择自定义用户名和密码。账号创建后，初始化模式永久关闭，除非手动重置数据库。

密码使用强密码哈希算法保存，例如 Argon2id 或 bcrypt。登录后，服务发放基于安全 cookie 的会话。生产部署文档应建议通过反向代理启用 HTTPS，并明确警告不要将服务直接暴露到公网。

## 主要模块

### `auth`

负责管理员初始化、登录、登出、会话、密码哈希和当前操作者查询。

### `roots`

负责已配置存储根目录、路径规范化、相对路径校验、绝对路径解析和根目录边界检查。

### `browser`

负责目录列表、文件元数据格式化、软链接元数据显示和自然排序。

### `rename`

负责 PowerRename 风格的规则解析、预览生成、冲突检测、枚举顺序、执行计划和撤销元数据。

### `ops`

负责文件操作，例如移动、复制、删除、创建软链接、创建硬链接和创建目录。

### `jobs`

负责 dry-run 计划、后台执行、进度跟踪、取消、失败记录和最终作业摘要。

### `audit`

负责持久记录谁在什么时候对哪些源路径和目标路径执行了什么操作。

### `web`

负责 REST 路由、请求校验、响应格式、错误映射和前端静态资源托管。

## 双栏文件管理器

主界面是双栏文件管理器。每一栏都可以选择任意已配置的存储根目录，并在该根目录内浏览目录。

每一栏支持：

- 显示当前根目录和路径。
- 目录导航。
- 文件和文件夹选择。
- 文件元数据显示。
- 自然排序。
- 刷新。

文件操作使用一栏作为源，另一栏作为目标。

第一版支持的操作：

- 移动选中项。
- 复制选中项。
- 在目标目录创建软链接。
- 在目标目录创建硬链接。
- 创建目录。
- 删除选中项。

批量操作使用同一个流程：

1. 用户选择源项。
2. 用户选择操作类型，并在需要时选择目标目录。
3. 前端发送 dry-run 请求。
4. 后端校验路径、评估冲突和权限，并返回操作计划。
5. 用户确认。
6. 后端创建作业并在后台执行。
7. 前端显示进度、失败明细和最终摘要。

## 批量重命名流程

批量重命名是独立工作流，可以从任意一栏中选中的文件或文件夹启动。

第一版应尽可能接近 PowerRename 行为，同时改进排序。

支持选项：

- 搜索文本。
- 替换文本。
- 正则表达式模式。
- 区分大小写匹配。
- 匹配所有出现位置。
- 应用于文件名、扩展名或两者。
- 包含文件。
- 包含文件夹。
- 包含子文件夹。
- 枚举项目。
- 预览原始名称和重命名后的名称。
- 执行前检测名称冲突。
- 作为后台作业执行。
- 为可逆重命名记录撤销元数据。

重命名预览必须使用与实际执行相同的操作计划。如果预览显示冲突或非法目标，执行时不能静默产生不同结果。

## 自然排序

FileButler 不能使用普通字典序对文件名排序。它应使用 Windows Explorer 风格的自然排序。

期望行为：

- 将名称拆分为数字片段和非数字片段。
- 非数字片段默认按不区分大小写比较。
- 数字片段按数值比较。
- 对数值相等的数字片段使用稳定的决胜规则，例如原始数字宽度和完整字符串比较。

示例顺序：

```text
file1
file2
file02
file10
file100
```

这确保 `02` 排在 `100` 前，符合用户对 Windows Explorer 排序的预期，也修正了规划阶段提到的 PowerRename 排序问题。

自然排序应用于：

- 文件列表显示。
- 批量重命名预览。
- 批量重命名枚举编号。
- 批量重命名执行顺序。
- 在顺序有意义时显示批量操作 dry-run 结果。

## 作业系统

所有批量操作都表示为作业。

作业阶段：

- 生成 dry-run 计划。
- 等待确认。
- 运行中。
- 已请求取消。
- 已完成。
- 已完成但有失败项。
- 失败。
- 已取消。

每个作业存储：

- 作业类型。
- 操作者 id。
- 源根目录和路径。
- 适用时的目标根目录和路径。
- 计划操作。
- 进度计数。
- 每一项的结果记录。
- 错误详情。
- 可用时的撤销元数据。
- 时间戳。

取消是协作式的。它只会在安全检查点停止，并不意味着回滚。已经完成的项目保持完成状态，并在最终结果中清楚展示。

## 撤销策略

第一版只在能安全表达的情况下支持撤销。

可撤销候选：

- 重命名操作。
- 同一文件系统内的移动。
- 由 FileButler 创建且未变化的软链接。
- 当目标可以被安全识别时，由 FileButler 创建的硬链接。

不保证完整撤销的操作：

- 复制操作。
- 跨文件系统移动，因为它可能表现为复制加删除。
- 取消前已部分完成的作业。
- 作业后被外部修改过的目标。

对于不可撤销的作业，FileButler 仍记录足够的审计信息用于人工清理，例如操作创建的目标路径。

## 冲突和错误处理

默认冲突策略是保守的：不覆盖已有文件。

dry-run 检测：

- 已存在的目标路径。
- 缺失的源路径。
- 执行前可见的权限错误。
- 根目录边界越界。
- 软链接边界问题。
- 跨文件系统硬链接限制。
- 非法重命名结果。
- 同一批次内重复的重命名目标。

执行时按项目记录结构化错误：

- 源路径。
- 适用时的目标路径。
- 操作类型。
- 错误分类。
- 系统错误消息。
- 有帮助时给出建议处理方式。

## 软链接和边界安全

已配置的根目录是主要安全边界。

后端必须：

- 清理相对路径。
- 解析绝对路径。
- 拒绝通过 `..` 遍历到根目录外。
- 校验解析后的路径仍位于已配置根目录内。
- 在写操作中谨慎处理软链接。

浏览时可以显示软链接目标元数据。如果写操作会导致 FileButler 跟随软链接操作到已配置根目录外，则不能跟随该软链接。

## API 形态

初始 API 分组：

- `GET /api/health`
- `GET /api/init/status`
- `POST /api/init/admin`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`
- `GET /api/roots`
- `GET /api/browse`
- `POST /api/ops/dry-run`
- `POST /api/ops/jobs`
- `POST /api/rename/preview`
- `POST /api/rename/jobs`
- `GET /api/jobs`
- `GET /api/jobs/{id}`
- `POST /api/jobs/{id}/cancel`
- `GET /api/audit`

准确的请求和响应 schema 将在实现计划中定义。

## 持久化

SQLite 存储：

- 管理员账号。
- 会话或会话引用。
- 存储根目录元数据快照。
- 作业。
- 作业项目结果。
- 审计记录。
- 撤销元数据。

第一版中，已配置根目录主要由配置文件驱动，而不是由用户在 UI 中创建。数据库可以在作业和审计记录中保存根目录快照，这样即使后续配置变化，历史日志仍然可理解。

## 部署

### Docker

Docker 镜像运行 Go 服务并托管构建后的前端。用户将宿主机目录挂载为已配置的存储根目录，并为 SQLite 数据库挂载持久化目录。

### Linux 本地

本地部署使用同一个二进制文件和配置文件。用户配置绝对路径形式的存储根目录，并选择数据库路径。
